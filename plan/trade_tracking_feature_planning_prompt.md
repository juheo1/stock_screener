# Prompt for Claude: Plan a Trade Tracking Page and Scanner-to-Tracker Workflow

Inspect the current stock screener repository and create a planning document only. Do not implement code yet.

Primary deliverable
- Create a planning document in the repo under `/plan/` for this feature. Choose an appropriate, specific filename.

Objective
Plan a new page and supporting data flow for tracking trades that the user actually executes based on signals from the Daily Strategy Scanner and related strategy/chart pages.

This feature is not only a static table. It should support:
- saving a scanner signal into a tracking sheet with relevant metadata,
- editing the actual executed trade details later,
- persisting data internally for the local user,
- importing/exporting CSV,
- and preserving enough quantitative metadata for later analysis.

This work must be planned in a codebase-specific way after inspecting the current repository.

## Core product concept

There should be a new page dedicated to tracking actual trade actions taken by the user.

The upstream source is the Daily Strategy Scanner. A user sees a detected signal, then chooses to add it to a trade-tracking sheet. The tracking sheet should preserve the original signal context while also allowing the user to record the actual execution that occurred in real life.

This means the feature must distinguish between:
- signal metadata generated by the system,
- market data at the time of signal,
- backtest metadata associated with that signal,
- and user-entered execution data.

## Required feature scope

### 1. Save from scanner into tracking sheet
From the Daily Strategy Scanner, there should be a clear action such as:
- Add to tracking sheet
- Track this signal
- Save trade candidate

When the user triggers this action, the tracking record should be created with all useful metadata copied from the signal context.

At minimum include:
- ticker
- signal side (buy/sell)
- strategy name
- strategy version if available
- strategy parameter snapshot if available
- scan date
- signal date
- signal category bucket the row came from
- source ETFs / source universe metadata if available
- signal detail / explanation payload if available
- backtest summary snapshot that was shown at the time
- chart timeframe
- open, high, low, close of the signal bar date
- close price of the signal date as the default suggested execution price
- any internal IDs needed to relate the trade-tracker row back to the scanner result

The plan must determine exactly where the scanner data already exists, how to pull it, and how to preserve reproducibility.

### 2. Editable actual execution fields
In the tracking sheet, the user must be able to adjust the actual execution.

The system-generated default should be based on the signal row, but the user must be able to override it because actual fills may differ from the signal close.

The plan should support both long and short workflows and should explicitly define field semantics.

The current idea is to unify the main executed price concept under one term such as:
- entry price / exit price
or
- execution price

Do not leave ambiguous terms like "strike price" if they are not correct for equities unless there is a strong reason in the codebase/UI.

At minimum, plan editable fields for:
- position side / action direction
- planned action from signal (buy/sell)
- actual entry date
- actual entry price
- actual exit date
- actual exit price
- quantity / shares
- notes
- execution status

The plan must support cases such as:
- user tracked a signal but did not actually enter
- user entered later than the signal date
- user partially entered but has not exited yet
- user edited an incorrect entry or exit
- user wants to mark a row as ignored / skipped / cancelled

### 3. Save / load / export
The tracking sheet must support:
- internal save/load for the local user
- export to CSV
- import from CSV

Storage expectations:
- save/load should use internal local persistence for the user, such as the local app DB/cache used by the project if available
- export should be plain CSV so the user can store it anywhere
- load should be able to read the internal stored tracking sheet
- optionally, the page should allow importing an external CSV by either:
  - drag/drop upload,
  - or file chooser / explorer navigation
- this should be cross-platform compatible: Windows, macOS, Linux

The plan must inspect the current app architecture and determine the best supported import/export approach in this stack.

### 4. Tracking sheet should include analysis-friendly fields
Think beyond a minimal order log. The tracking sheet should include columns useful for later quantitative analysis.

Recommend additional columns and derived metrics that should be captured or computed, such as:
- signal close price
- next open price relative to signal close
- gap amount / gap percent
- slippage from signal price to actual entry
- holding period in calendar days and trading days
- realized P&L
- unrealized P&L if still open
- return percent
- MFE / MAE if feasible using available data
- win/loss status
- strategy bucket
- setup quality or signal score if present in scanner output
- reason for exit
- whether the trade followed the original signal side or was discretionary
- whether the execution was same-day, next-day, or delayed
- whether the signal became stale before execution
- tags / labels for later filtering

The plan should separate:
- raw stored fields,
- editable user-entered fields,
- derived/calculated fields.

### 5. Page behavior and UX
The new page should function as a trade-tracking workspace, not just a static table.

Plan the UX for:
- table view of tracked rows
- selecting a row to inspect details
- inline editing vs side-panel editing vs modal editing
- filtering and sorting
- save state indicators
- import/export actions
- linking back to scanner result and/or chart view
- handling open vs closed positions

The plan should consider useful views such as:
- all tracked items
- open positions
- closed positions
- skipped / ignored signals
- grouped by strategy
- grouped by ticker

### 6. Relationship to existing features
This page must be planned to integrate with:
- Daily Strategy Scanner
- Technical Chart
- existing strategy definitions
- existing backtest summary payloads
- any existing persistence/storage layer in the project

When possible, the plan should reuse:
- existing IDs and schemas
- existing chart drill-down flow
- existing strategy metadata
- existing result payloads
- existing local persistence utilities

### 7. Data model requirements
The plan must define a clear schema for the tracking record.

Separate the schema conceptually into:
- source signal snapshot
- market snapshot at signal time
- strategy/backtest snapshot
- user execution data
- derived analytics fields
- audit / metadata fields

At minimum, include recommendations for:
- primary key
- created_at / updated_at timestamps
- source scanner job ID if relevant
- source signal ID if available
- source strategy name and parameter snapshot
- instrument symbol
- asset type if relevant
- timeframe
- signal side
- signal date
- signal OHLC
- suggested entry price
- user entry date/price
- user exit date/price
- quantity
- cost basis
- status enum
- user notes
- tags

### 8. CSV import/export requirements
The plan should define a robust CSV schema and import rules.

Need to address:
- required columns
- optional columns
- stable column names
- version column if useful
- validation rules
- import conflict handling
- duplicate-row handling
- whether import should merge, replace, or append
- how to handle missing derived fields

### 9. Additional helpful features to consider in scope planning
Recommend additional helpful features if they materially improve the tracker. Examples to consider:
- open/closed/skipped status enum
- planned vs actual execution comparison
- link to open the associated chart with the same strategy/indicator bundle
- quick mark-as-entered / mark-as-exited actions
- bulk export
- bulk tagging
- derived dashboard summaries later
- optional commission/fee field
- optional stop-loss / take-profit plan fields
- optional risk-per-trade field
- optional portfolio/account label
- optional timezone handling for dates

Do not over-scope implementation, but include useful recommendations and clearly separate V1 vs later enhancements.
